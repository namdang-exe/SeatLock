## Section 2 ‚Äî Core Entities ‚úÖ ALL RESOLVED

### Venue

| # | Status | Question | Resolution |

### Slot

| # | Status | Question | Resolution |
|---|--------|----------|------------|
| OQ-04 | ‚úÖ | Fixed slot duration or admin-defined per slot? | **Configurable system constant** ‚Äî `seatlock.slot.duration-minutes=60`. Currently 60 minutes. Only startTime stored; endTime is derived. |
| OQ-05 | ‚úÖ | Recurring auto-generated or individually created? | **Recurring rule** ‚Äî Mon‚ÄìFri, 9am‚Äì5pm, 1h blocks. Auto-generated by venue-service. |
| OQ-06 | ‚úÖ | Slot state machine ‚Äî is AVAILABLE‚ÜíHELD‚ÜíBOOKED‚ÜíCANCELLED correct? | **Corrected** ‚Äî no CANCELLED on Slot. Final machine: `AVAILABLE ‚Üí HELD ‚Üí BOOKED ‚Üí AVAILABLE` (via cancellation). CANCELLED belongs on Booking, not Slot. |

### Hold

| # | Status | Question | Resolution |
|---|--------|----------|------------|
| OQ-07 | ‚úÖ | What data does a Hold carry beyond (userId, slotId, expiresAt)? | **holdId, sessionId, userId, slotId, expiresAt, status**. sessionId groups multi-slot holds in one transaction. |
| OQ-08 | ‚úÖ | One Hold per transaction or one per slot? | **One Hold per slot** with shared sessionId. Each slot's Redis TTL is independent; sessionId provides grouping. |
| OQ-09 | ‚úÖ | After confirmation ‚Äî Hold deleted or terminal state? | **Both**: Redis key deleted (keeps Redis lean); Postgres Hold status ‚Üí CONFIRMED (audit trail). Booking stores holdId as reference. |

## Section 3 ‚Äî API / Interface ‚úÖ SHAPES RESOLVED

| # | Status | Question | Resolution |
|---|--------|----------|------------|
| OQ-10 | ‚úÖ | Real-time availability ‚Äî polling vs. SSE vs. WebSockets? | **RESOLVED (Section 4):** Frontend polls every 5s, matching the 5s Redis availability cache TTL. SSE and WebSockets are overkill for read-only status updates at 10,000 DAU scale. |
| OQ-11 | ‚úÖ (partial) | Idempotency strategy for hold creation and booking confirmation? | **Partially resolved in Phase 0:** `POST /holds` requires `Idempotency-Key` header; value becomes `sessionId`; duplicate ‚Üí return existing ACTIVE holds. `POST /bookings` checks for existing CONFIRMED bookings by `sessionId` at step 0; duplicate ‚Üí return 201 with existing data. Full Redis-backed idempotency store with TTL-keyed response cache remains Phase 1. |
| OQ-12 | ‚úÖ | API versioning strategy? | **RESOLVED (Section 4):** URL path prefix `/api/v1/` ‚Äî simple, already adopted in all endpoint shapes. Header-based versioning deferred to a future API version. |
| OQ-13 | ‚úÖ | What are the exact request/response shapes for: browse slots, create hold, confirm booking, cancel booking? | **Resolved** ‚Äî full shapes defined in `docs/system-design/03-api-interface.md`. All critical paths covered: POST /holds, POST /bookings (201), POST /bookings/{confirmationNumber}/cancel, GET /bookings, GET /venues/{venueId}/slots. |
| OQ-14 | ‚úÖ (partial) | Rate limiting ‚Äî which endpoints need it, and what are the limits? | **Partially resolved** ‚Äî `POST /holds` identified as minimum rate-limited endpoint (prevents slot squatting). Specific numeric limits (e.g. requests/user/minute) not yet set; not blocking for Phase 0 design. |
