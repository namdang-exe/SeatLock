# ADR-004: availability-service Merged into venue-service

## Status
Accepted

## Date
2026-02-24

## Context

The initial design included a dedicated availability-service responsible for reading slot availability from Redis cache and falling back to Postgres on cache miss.

Availability data is a read view over the `slots` table, which is owned by venue-service. To serve availability reads, availability-service would need access to slot data — either by sharing venue-service's database (violates service isolation) or by making a synchronous HTTP call to venue-service on every cache miss (adds a hop on the hot read path).

The availability read path is the hottest path in the system: the frontend polls every 5 seconds per active user. At 200 peak concurrent users, this is ~40 requests/second of cache reads. The p99 latency target is <200ms.

## Decision

availability-service is merged into venue-service. venue-service now owns: venue CRUD, slot CRUD and auto-generation, slot reads, and availability reads (Redis cache with Postgres fallback).

booking-service continues to own cache invalidation (`DEL slots:{venueId}:{date}`) on every slot status write.

## Consequences

**Positive:**
- Cache miss → Postgres fallback is a direct DB call with no extra network hop
- No shared-DB violation — venue-service reads its own schema
- Eliminates a service, its deployment, and its failure surface
- Keeps the hot read path within a single service boundary

**Negative:**
- venue-service's responsibilities are broader than a pure venue catalog service
- Cache invalidation is split: venue-service owns cache reads; booking-service owns cache writes (DEL) — requires clear documentation of this ownership boundary

## Alternatives Rejected

| Alternative | Reason rejected |
|-------------|----------------|
| Separate availability-service with shared DB | Violates the core microservice principle that each service owns its data |
| Separate availability-service calling venue-service | Adds a synchronous HTTP hop on every cache miss on the highest-frequency read path; breaks the <200ms latency target under cache pressure |
