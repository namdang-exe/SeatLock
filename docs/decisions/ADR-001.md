# ADR-001: Microservices Architecture

## Status
Accepted

## Date
2026-02-27

## Context

SeatLock is a distributed reservation platform built as a learning project to demonstrate production-grade distributed systems design. The architecture must demonstrate service isolation, independent deployability, and resilience patterns — not just functional correctness.

The core domains are: user identity, venue and slot management, booking lifecycle, and notification dispatch. These have distinct data ownership, different scaling needs, and different failure characteristics.

## Decision

SeatLock is built as a microservices system. Each service owns its own database schema, is independently deployable, and maps to a single bounded context.

The four services are:
- **user-service** — registration, login, JWT issuance
- **venue-service** — venue CRUD, slot management, availability reads
- **booking-service** — hold creation, confirmation, cancellation, expiry
- **notification-service** — email, SMS, and in-app dispatch

## Consequences

**Positive:**
- Each service can be deployed, scaled, and restarted independently
- Failures in one service do not cascade to others (with proper circuit breakers)
- Service boundaries enforce clean domain separation
- Learning objectives for distributed systems patterns (service discovery, async messaging, inter-service auth, degraded modes) are fully exercised

**Negative:**
- More operational complexity than a monolith (service discovery, inter-service auth, distributed tracing)
- Phase 0 compromise: booking-service and venue-service share a Postgres cluster (see ADR-006) — true DB isolation is deferred

## Alternatives Rejected

| Alternative | Reason rejected |
|-------------|----------------|
| Modular monolith | Simpler to operate but does not exercise the distributed systems patterns that are the primary learning objective |
| Serverless functions per endpoint | Eliminates persistent state management (Redis connection pooling, scheduled jobs) and adds cold-start latency incompatible with the <500ms hold latency target |
