# ADR-003: booking-service Owns Hold Lifecycle (hold-service Merged)

## Status
Accepted

## Date
2026-02-27

## Context

The initial design considered a dedicated hold-service responsible for creating and expiring holds, with booking-service responsible for converting holds into confirmed bookings.

The booking confirmation flow requires: validate hold, write booking record, update slot status, update hold status — atomically. If hold-service and booking-service are separate, the confirmation path requires a synchronous cross-service call on every booking confirmation.

## Decision

hold-service is merged into booking-service. booking-service owns the full reservation lifecycle: hold creation, booking confirmation, cancellation, booking history, and hold expiry.

## Consequences

**Positive:**
- Booking confirmation is a single in-process operation — no cross-service call on the critical path
- The `INSERT INTO bookings` and `UPDATE holds SET status = 'CONFIRMED'` execute in the same Postgres transaction — atomicity is trivial
- Reduces service count, operational overhead, and inter-service failure surface

**Negative:**
- booking-service is the largest service by responsibility; it owns holds, bookings, the expiry job, Redis cache invalidation, and slot.status writes
- If the hold domain grows significantly (e.g. pricing holds, multi-party holds), extraction into a separate service becomes desirable

## Alternatives Rejected

| Alternative | Reason rejected |
|-------------|----------------|
| Separate hold-service | Booking confirmation requires a synchronous hold-service call; if hold-service is slow or unavailable, booking confirmations fail; cross-service call adds latency on the critical path |
