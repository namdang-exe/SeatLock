# ADR-006: Shared Postgres Cluster (Phase 0 Pragmatic Compromise)

## Status
Accepted (Phase 0 only) — to be revisited in Phase 1

## Date
2026-02-25

## Context

booking-service must write `slots.status` as part of its hold creation, booking confirmation, and cancellation transactions. This write must be atomic with the corresponding `holds` and `bookings` table writes — a secondary safety gate (`AND status = 'AVAILABLE'`) depends on this atomicity.

In a true microservices architecture, venue-service would own the `slots` table exclusively. booking-service would need to update slot status via an API call to venue-service or an async event. However:

- An async event would decouple `slots.status` from the Postgres transaction, breaking the secondary safety gate for the ≤60s expiry lag window
- A synchronous API call would add a network hop inside the booking transaction, creating a distributed transaction problem

## Decision

In Phase 0, booking-service and venue-service share the same Postgres cluster. Service ownership is enforced at the application layer:

- venue-service owns all `slots` columns **except** `status`
- booking-service writes `slots.status` only, inside its own transactions
- No cross-schema joins; each service's tables are logically separated

This is a documented architectural compromise, not a permanent design.

## Consequences

**Positive:**
- `slots.status` UPDATE is atomic with `holds` INSERT and `bookings` INSERT — no distributed transaction needed
- The secondary Postgres safety gate (`AND status = 'AVAILABLE'`) is enforced without cross-service coordination
- Simplifies Phase 0 implementation significantly

**Negative:**
- Violates the microservice principle that each service owns its database schema
- A schema migration to `slots` requires coordination between both services
- True DB isolation (extracting slot availability state into booking-service's schema) is a Phase 1+ goal

## Phase 1 Migration Path

Extract a `slot_availability` table (or schema) into booking-service's database. venue-service publishes slot metadata events; booking-service maintains its own availability projection. This removes the shared cluster dependency.

## Alternatives Rejected

| Alternative | Reason rejected |
|-------------|----------------|
| Async event for slot.status update | Breaks atomicity of the hold transaction; secondary Postgres guard cannot be enforced if slot.status is updated asynchronously |
| Synchronous API call to venue-service inside Postgres transaction | Creates a distributed transaction (two-phase commit) — adds failure modes, latency, and complexity disproportionate to Phase 0 scope |
