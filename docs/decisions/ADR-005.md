# ADR-005: Async SQS for booking-service → notification-service

## Status
Accepted

## Date
2026-02-25

## Context

Three booking lifecycle events require user notification: hold expired, booking confirmed, and booking cancelled. notification-service is responsible for dispatching email, SMS, and in-app messages.

The question is whether booking-service should call notification-service synchronously (HTTP) or publish events asynchronously (SQS).

The user's HTTP response for `POST /bookings` and `POST /bookings/{id}/cancel` must not depend on whether notifications are delivered. Notification delivery is a downstream concern with its own failure modes (email provider rate limits, SMS delivery delays).

## Decision

booking-service publishes events to an SQS Standard Queue after each lifecycle transition. notification-service is a stateless SQS consumer that dispatches messages on receipt.

Events published:
- `BookingConfirmedEvent` — after `POST /bookings` Postgres commit
- `BookingCancelledEvent` — after `POST /bookings/{confirmationNumber}/cancel` Postgres commit
- `HoldExpiredEvent` — after the `@Scheduled` expiry job marks holds as EXPIRED

## Consequences

**Positive:**
- notification-service availability is decoupled from booking write path — a degraded notification-service does not fail or slow booking confirmations
- Events are durable in SQS — delivery is retried automatically if notification-service is temporarily unavailable
- notification-service is stateless and independently scalable

**Negative:**
- Notification delivery is eventually consistent — there is a short lag between booking action and notification receipt
- Adds SQS as a dependency; requires dead-letter queue configuration for poison messages (Phase 1)

## Alternatives Rejected

| Alternative | Reason rejected |
|-------------|----------------|
| Synchronous HTTP to notification-service | notification-service availability becomes a hard dependency of every booking write; a degraded notification provider (e.g. SendGrid outage) would fail booking confirmations |
| In-process notification dispatch | Tightly couples notification logic to booking-service; blocks the booking HTTP response on delivery; makes notification-service unscalable independently |
