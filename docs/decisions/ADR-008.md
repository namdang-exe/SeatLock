# ADR-008: Postgres-Before-Redis-DEL Operation Order in POST /bookings

## Status
Accepted

## Date
2026-02-27

## Context

The booking confirmation flow (`POST /bookings`) must atomically convert holds into confirmed bookings. The original operation order was:

```
1. Validate Redis hold keys
2. DEL Redis hold keys
3. Postgres transaction (INSERT bookings, UPDATE slots, UPDATE holds)
```

If booking-service crashes between steps 2 and 3, Redis keys are gone but no booking record exists in Postgres. On retry, the user gets 409 `HOLD_EXPIRED` (keys missing) even though the booking was never created. Recovery requires manual intervention or a saga compensating transaction.

A secondary consequence: crash-induced stale Redis hold keys (where DEL was skipped) remain in Redis until TTL expiry. If the booking is later cancelled and a new user attempts to hold the same slot, `SETNX` fails on the stale key — the slot appears unavailable even though it is genuinely free.

## Decision

**Reorder operations: Postgres commits before Redis DEL.**

```
1. Validate Redis hold keys
2. Postgres transaction (INSERT bookings, UPDATE slots, UPDATE holds) ← source of truth
3. DEL Redis hold keys ← best-effort cleanup
```

**Additionally: add `DEL hold:{slotId}` to the cancellation flow.**

The cancellation flow already issues `DEL slots:{venueId}:{date}`. Adding `DEL hold:{slotId}` for each slot clears any crash-induced stale hold key. `DEL` on a missing key is a no-op — this call is always safe.

## Consequences

**Positive:**
- Crash before Postgres commits: no side effects; retry sees Redis keys intact and proceeds normally
- Crash after Postgres commits: retry hits step 0 idempotency check, finds existing CONFIRMED bookings, returns 201 — no re-execution
- No saga, no outbox, no distributed transaction required
- Stale hold key problem after cancellation is eliminated

**Negative:**
- DEL is now best-effort; stale Redis hold keys may exist briefly if the service crashes after Postgres commits
- Stale keys are harmless for availability correctness (slot is BOOKED in Postgres; `AND status='AVAILABLE'` guard on new hold attempts catches this) but the cancellation-path DEL is required to clear them when the slot is released

## Alternatives Rejected

| Alternative | Reason rejected |
|-------------|----------------|
| Saga / compensating transactions | Adds distributed coordination complexity; the reorder achieves the same crash safety with zero additional infrastructure |
| Outbox pattern | Adds an outbox table, a background relay process, and at-least-once delivery semantics; unnecessary when the simpler reorder eliminates the failure window entirely |
| Accept and rely on TTL cleanup | Stale keys block new holds on a freed slot for up to 30 minutes after cancellation — unacceptable UX |
