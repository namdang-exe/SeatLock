# ADR-002: Redis SETNX as Primary Concurrency Gate

## Status
Accepted

## Date
2026-02-27

## Context

SeatLock has a zero-tolerance requirement for double-bookings. Multiple users may attempt to hold the same slot simultaneously. The system must guarantee that exactly one hold wins per slot, without holding a database lock for the duration of the user's decision window (up to 30 minutes).

The peak hold creation rate is ~50 holds/minute. The p99 latency target for hold creation is <500ms.

## Decision

Redis `SETNX hold:{slotId} {holdId} EX 1800` is the primary double-booking gate.

- `SETNX` (SET if Not eXists) is atomic — exactly one concurrent caller wins per key
- The `EX 1800` (30-minute TTL) makes the hold self-expiring without requiring the application to clean up explicitly
- The call is sub-millisecond — orders of magnitude faster than a Postgres row lock
- Only the SETNX winner proceeds to the Postgres write; losers get 409 `SLOT_NOT_AVAILABLE` immediately

A secondary safety gate (`AND status = 'AVAILABLE'` on the Postgres `UPDATE slots`) catches the ≤60s expiry lag window where a Redis key has expired but the Postgres record has not yet been updated by the `@Scheduled` expiry job.

## Consequences

**Positive:**
- Prevents double-booking without holding a Postgres row lock for 30 minutes
- Sub-millisecond atomic operation keeps the hold path well within the 500ms p99 target
- Native TTL aligns with the 30-minute hold duration with no extra cleanup logic

**Negative:**
- Redis is now a hard runtime dependency for booking writes (hold creation fails if Redis is down)
- Resolved by OQ-21: system enters read-only degraded mode when Redis is unavailable (see ADR-005)

## Alternatives Rejected

| Alternative | Reason rejected |
|-------------|----------------|
| Postgres `SELECT FOR UPDATE` | Holds a row lock for up to 30 minutes while the user decides; at 50 holds/min peak this would cause severe lock contention on the slots table |
| Optimistic locking (version field) | Requires retry on conflict; under high concurrency (same slot), most retries fail, generating thundering-herd load on Postgres |
| Application-level mutex | Does not work across multiple ECS task instances; not horizontally scalable |
