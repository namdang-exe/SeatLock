# ADR-007: Service JWT for Inter-Service Authentication

## Status
Accepted

## Date
2026-02-25

## Context

booking-service makes synchronous HTTP calls to venue-service (slot existence verification at hold creation). These calls must be authenticated — an unauthorized process inside the VPC must not be able to call internal APIs.

The `@Scheduled` expiry job in booking-service also calls venue-service. It has no user context and cannot forward a user JWT.

## Decision

booking-service signs a short-lived JWT with a service identity claim, using a shared HMAC secret sourced from Vault at startup:

```json
{
  "sub": "booking-service",
  "iss": "seatlock-internal",
  "exp": <now + 5 minutes>
}
```

This token is placed in the `Authorization: Bearer` header of every internal call to venue-service. venue-service validates the JWT signature (using the same Vault-sourced shared secret) and the `sub` claim before processing the request.

Both services retrieve the shared signing secret from Vault on startup.

## Consequences

**Positive:**
- Works for all callers including the expiry job — no user context required
- Defense-in-depth beyond network-level security groups — unauthorized processes inside the VPC cannot call venue-service without the credential
- Reuses existing Spring Security JWT infrastructure already in the stack — no new libraries
- Caller identity (`sub: booking-service`) is explicit in logs and audit trails

**Negative:**
- Shared HMAC secret means any holder of the secret can both sign and verify tokens — slightly weaker than asymmetric signing (RS256)
- Secret rotation requires a coordinated restart of both services (or a grace period with two valid secrets)
- Scales poorly beyond a handful of services — a service mesh with mTLS is more appropriate at higher scale

## Phase 1+ Consideration

Asymmetric JWT (RS256): booking-service holds a private key; venue-service holds only the public key. A holder of the public key cannot forge tokens. More appropriate if the number of internal callers grows.

## Alternatives Rejected

| Alternative | Reason rejected |
|-------------|----------------|
| Forward user JWT | Expiry job has no user context; ties internal trust to user session lifecycle, which is semantically incorrect |
| IAM task roles + SigV4 request signing | Correct AWS-native approach for AWS service calls but adds SDK signing complexity for internal HTTP; better suited for S3, DynamoDB, etc. |
| Network-level trust only (security groups) | No application-level identity; a compromised process in the same security group could call internal APIs without restriction |
| mTLS | Strong mutual authentication but adds certificate lifecycle management (issuance, rotation, revocation) — overkill for 4 services in Phase 0 |
