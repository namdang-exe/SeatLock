# 02 — Core Entities

## Context

This document defines the five core domain objects in SeatLock, their lifecycles, state machines, field ownership, and the service responsible for each. These decisions were made in the Phase 0 design interview.

---

## Entity Overview

| Entity | Source of Truth | Owned By |
|--------|----------------|----------|
| User | Postgres | user-service |
| Venue | Postgres | venue-service |
| Slot | Postgres | venue-service |
| Hold | Redis (TTL lock) + Postgres (audit) | booking-service |
| Booking | Postgres | booking-service |

---

## Entity 1 — User

### What it owns
A registered account with a persistent identity used for auth, hold tracking, and notification delivery.

### Key fields
| Field | Type | Notes |
|-------|------|-------|
| `userId` | UUID | Primary key |
| `email` | String | Unique, used for JWT identity + email notifications |
| `passwordHash` | String | Bcrypt |
| `phone` | String | For SMS notifications |
| `createdAt` | Timestamp | |

### Lifecycle
Users register once. No guest booking. JWT is issued on login and used to authenticate all subsequent requests.

### Decisions
- We decided **no guest booking** because holds and notifications require a persistent identity.

## Entity 2 — Venue

### What it owns
The bookable resource — a room, court, desk, or any physical space. A Venue has many Slots.

### Key fields
| Field | Type | Notes |
|-------|------|-------|
| `venueId` | UUID | Primary key |
| `name` | String | Display name |
| `address` | String | Street address |
| `city` | String | |
| `state` | String | |
| `status` | Enum | `ACTIVE` / `INACTIVE` |
| `createdAt` | Timestamp | |

### Lifecycle & State Machine
```
ACTIVE → INACTIVE   (admin deactivates — soft-delete)
INACTIVE → ACTIVE   (admin reactivates)
```

Deactivating a Venue does **not** delete it or its Slots/Bookings. Historical data is preserved. An `INACTIVE` venue does not appear in availability queries.

### Decisions
- We decided on **soft-delete** (`ACTIVE`/`INACTIVE`) because deleting a Venue with historical bookings would break referential integrity and remove audit history.
- We decided **capacity is always 1** — one booking per slot, entire venue. No per-venue capacity configuration. This keeps the concurrency model simple.
- We decided **admin slot disable is out of scope** for Phase 0.

### Alternatives Considered
- **Hard-delete:** Rejected. Would cascade-delete Slots and Bookings, destroying historical records.
- **Configurable capacity per slot:** Rejected for Phase 0. Capacity > 1 would require a quantity field on Hold and Booking, and the concurrency gate would need to track remaining capacity rather than a binary lock.

## Entity 3 — Slot

### What it owns
A specific time window on a Venue that can be held and booked. A Slot is the atomic bookable unit.

### Key fields
| Field | Type | Notes |
|-------|------|-------|
| `slotId` | UUID | Primary key |
| `venueId` | UUID | FK to Venue |
| `startTime` | Timestamp | Start of the bookable window |
| `endTime` | Timestamp | Derived: `startTime + slot.duration-minutes` |
| `status` | Enum | `AVAILABLE` / `HELD` / `BOOKED` |
| `createdAt` | Timestamp | |

### Slot Duration
Slot duration is a **configurable system constant**, not stored per slot:
```yaml
seatlock.slot.duration-minutes: 60
```
Current value: **60 minutes**. All slots are exactly 1 hour. Only `startTime` is stored; `endTime` is derived.

### Slot Generation
Slots are **auto-generated by a recurring rule**: Monday–Friday, 9am–5pm, in 1-hour blocks. The admin defines the rule per venue; the system generates the individual Slot records. Slots are not created manually one-by-one.

### State Machine
```
AVAILABLE ──(user places hold)──────────► HELD
HELD      ──(hold expires)──────────────► AVAILABLE
HELD      ──(user abandons / releases)──► AVAILABLE
HELD      ──(user confirms booking)─────► BOOKED
BOOKED    ──(user cancels, > 24h before start)──► AVAILABLE
```

### Key Rules
- A Slot can only be `HELD` by one user at a time (enforced by Redis SETNX — see `06-deep-dives.md`).
- When a Booking is cancelled, the Slot immediately returns to `AVAILABLE` and is re-bookable by anyone.
- Cancellation is only permitted if the current time is **more than 24 hours before** `slot.startTime`.
- There is no `CANCELLED` state on Slot. Cancellation history lives on the Booking record.

### Alternatives Considered
- **`CANCELLED` state on Slot:** Rejected. A cancelled Slot is just an `AVAILABLE` Slot — the cancellation event belongs on the Booking. Adding `CANCELLED` to Slot would conflate availability status with booking history.
- **Fixed slot duration (hardcoded):** Rejected. Using a system constant allows the value to be changed via config without a code deploy.
- **Manual slot creation:** Rejected in favor of recurring rules. Manual creation would require admins to create hundreds of individual slots, which is impractical.

## Entity 4 — Hold

### What it owns
A 30-minute reservation lock on a Slot. Prevents double-booking while a user is mid-flow. The Hold is the concurrency gate.

### Key fields
| Field | Type | Notes |
|-------|------|-------|
| `holdId` | UUID | Primary key |
| `sessionId` | UUID | Groups multiple Holds in one user transaction |
| `userId` | UUID | FK to User |
| `slotId` | UUID | FK to Slot |
| `expiresAt` | Timestamp | `now + 30 minutes` |
| `status` | Enum | `ACTIVE` / `CONFIRMED` / `EXPIRED` / `RELEASED` |
| `createdAt` | Timestamp | |

### Where it lives
| Store | What it holds | Why |
|-------|--------------|-----|
| **Redis** | `hold:{slotId}` → `{holdId, userId, sessionId, expiresAt}` with TTL = 30min | Atomic SETNX is the concurrency gate; TTL handles automatic expiry |
| **Postgres** | Full Hold record with `status` field | Audit trail, queryability, grouping by `sessionId` |

### SessionId
When a user starts a multi-slot flow, a `sessionId` (UUID) is generated. Every Hold in that flow shares the same `sessionId`. This allows:
- Independent TTL per slot in Redis (each slot's hold expires on its own)
- Grouping all holds in a transaction for confirmation or abandonment

### State Machine
```
ACTIVE ──(user confirms booking)──► CONFIRMED
ACTIVE ──(TTL expires in Redis)───► EXPIRED
ACTIVE ──(user releases hold)─────► RELEASED
```

### After Confirmation
On booking confirmation:
1. Redis key `hold:{slotId}` is deleted (keeps Redis lean)
2. Postgres Hold record status → `CONFIRMED`
3. Booking record is created with a reference to `holdId`

### Decisions
- We decided **one Hold record per slot** (not one per transaction) because each slot's TTL must be managed independently in Redis. A slot's hold expires regardless of what happens to other slots in the same session.
- We decided to use **sessionId** to group multi-slot holds, giving the best of both approaches: independent expiry + transactional grouping.
- We decided **booking-service owns Hold** because Hold and Booking are the same domain (reservation lifecycle). Splitting them into hold-service + booking-service would introduce a synchronous cross-service call on the critical confirmation path.
- We decided to **delete the Redis key on confirmation** and **update Postgres status to `CONFIRMED`** so Redis stays lean while the full audit trail lives in Postgres.

### Alternatives Considered
- **One Hold record per transaction:** Rejected. A single TTL cannot represent multiple slots expiring at the same moment if they were held at different times. Independent per-slot TTLs in Redis are cleaner.
- **Hold-service as a separate service:** Rejected. Would require booking-service to call hold-service synchronously during confirmation — an extra network hop on the most latency-sensitive path.
- **Keep Hold in Redis after confirmation:** Rejected. Redis is not an audit store. Confirmation history belongs in Postgres.

## Entity 5 — Booking

### What it owns
The confirmed, committed reservation. The terminal record of a successful hold-to-confirm flow.

### Key fields
| Field | Type | Notes |
|-------|------|-------|
| `bookingId` | UUID | Primary key |
| `sessionId` | UUID | Groups all Booking records in the same transaction |
| `confirmationNumber` | String | Human-readable (e.g., `SL-20240218-4821`) — **same value for every Booking in the session** |
| `userId` | UUID | FK to User |
| `slotId` | UUID | FK to Slot |
| `holdId` | UUID | FK to Hold (audit — traces booking back to its hold) |
| `status` | Enum | `CONFIRMED` / `CANCELLED` |
| `createdAt` | Timestamp | |
| `cancelledAt` | Timestamp | Nullable |

### State Machine
```
CONFIRMED ──(user cancels, > 24h before slot start)──► CANCELLED
```

No other transitions. There is no `PENDING` state because payment is out of scope — confirmation is the terminal action.

### Confirmation Number Format
Human-readable reference for users to quote to venue operators:
`SL-{YYYYMMDD}-{random-4-digit-suffix}` — e.g., `SL-20240218-4821`.

Generated **once per session** at `POST /bookings` time by booking-service.
Stored redundantly on **every Booking record in the session** so any single record can answer "what session does this belong to?" without a join.
One user-facing confirmation number therefore represents one or more booked slots.

### Cancellation Rule
Cancellation operates at the **session level**, not the individual booking level.
`POST /bookings/{confirmationNumber}/cancel` cancels **all** Booking records that share that `confirmationNumber` in a single atomic transaction.

Each booking can only be cancelled if:
```
currentTime < slot.startTime - 24 hours
```
The 24h rule applies to **all slots in the session collectively**: if any slot in the session violates the window, the entire cancellation is rejected with a `409 Conflict`.
On success, every affected Slot is immediately returned to `AVAILABLE`.

### Decisions
- We decided to include a **human-readable confirmation number** because users need to quote it to venue operators in real-world interactions.
- We decided the Booking state machine is **`CONFIRMED → CANCELLED` only** — no `PENDING` state — because payment is out of scope and confirmation is immediate.
- We decided to store `holdId` on the Booking as an **audit reference** so the hold-to-booking journey can be traced without keeping stale data in Redis.
- We decided **one `confirmationNumber` per session** (not per Booking record) because users expect a single reference that covers all slots they booked together; a per-record number would mean a user juggling multiple codes for one transaction.
- We decided `sessionId` is stored on every Booking record (redundantly) so a query by `confirmationNumber` returns all sibling bookings without an extra join through Hold.
- We decided **session-level cancellation** (cancel by `confirmationNumber`) because a user booking multiple slots together expects them to cancel as a unit; partial cancellation of a session is not supported in Phase 0.
- We decided the **dual-key pattern** (`sessionId` + `confirmationNumber`) on every Booking row is intentional denormalization. Each key serves a distinct semantic role: `sessionId` (UUID) is the internal transaction identity that spans the entire hold → confirm lifecycle and is used by booking-service internals and the `POST /bookings` request body; `confirmationNumber` (human-readable `SL-YYYYMMDD-XXXX`) is the user-facing booking reference that appears in emails, receipts, cancel URLs, and venue operator lookups. Before confirmation, `sessionId` exists but `confirmationNumber` does not (it is generated at `POST /bookings` time). After confirmation they are bijective — one can always be looked up from the other — but storing both as indexed columns on every Booking row avoids a join through the Hold table and allows either identifier to directly answer "which session does this booking belong to?". This is the same surrogate + natural key pattern used in e-commerce (`order_id` + `order_number`) and ITSM (`incident_id` + `ticket_number`).
